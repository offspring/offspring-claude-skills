# Repo Documentation Setup — Template

Set up standardized documentation for the current repo. Safe to re-run — existing content is preserved and merged, not replaced.

> **Before you start:** Ensure all existing changes in this repo are committed. This template will consolidate and may delete documentation files. If something goes wrong, you'll need `git checkout` or `git diff` to recover. Run `git status` first and commit or stash any uncommitted work.

## Step 1: Explore the codebase

Before creating anything, thoroughly explore this repo to understand:

- What the service/library does (read existing README, main entrypoints)
- All external integrations (APIs called, APIs exposed, message queues, databases)
- Key internal components and how they connect
- Build system (package.json scripts, Makefile targets, etc.)
- Test setup (unit, integration, e2e)
- Authentication/authorization mechanisms
- Configuration (flags, env vars, config files)
- Code generation (if any — OpenAPI, protobuf, GraphQL, mocks, etc.)
- Existing documentation (any .md files, docs/ directories)

## Step 2: Consolidate existing docs

Every repo must end up with exactly these top-level docs and nothing else:

| File | Purpose | Owns |
| ---- | ------- | ---- |
| `README.md` | Landing page | About, badges, local dev setup, test instructions, doc links |
| `ARCHITECTURE.md` | Technical reference | API endpoints, integrations, data flows, data models, config, security, metrics, references |
| `DEVELOPER_GUIDE.md` | Operational guide | Troubleshooting, auth/secrets setup, API examples, enhancement ideas |
| `CLAUDE.md` | Claude Code context (root) | Build commands, project structure, conventions |

Before creating new docs, inventory ALL existing documentation in the repo:

1. Scan for all `.md` files: root, `docs/`, `doc/`, any other directories
2. For each existing doc, classify its content into one of the 4 target docs above
3. Plan the migration — map each section of each existing doc to its target:
   - API docs, integration docs, data model docs, swagger/OpenAPI references → **ARCHITECTURE.md**
   - Runbooks, troubleshooting guides, auth/secrets setup, curl examples, operational procedures → **DEVELOPER_GUIDE.md**
   - Design docs links, meeting notes links, external references → **ARCHITECTURE.md** (References section)
   - Build instructions, project structure, conventions → **CLAUDE.md**
   - About/overview, dev setup, test instructions → **README.md**
4. Content that doesn't fit any target doc should be evaluated: if it's stale or obsolete, discard it. If it's still valuable, find the closest target doc.

After creating the new docs (Steps 3-6), **present the user with a list of files to be deleted and ask for explicit approval before deleting anything.** Do NOT delete any documentation files without user confirmation.

Files that are candidates for deletion after consolidation:

- `docs/` or `doc/` directories — **unless** the directory contains a hosted documentation build config (see exception below) or holds only diagram files under `docs/diagrams/` (created in Step 4)
- Standalone docs like `API.md`, `INTEGRATION.md`, `DATA_MODEL.md`, `CONTRIBUTING.md` (merge relevant content first)
- Do NOT delete non-doc files (e.g. `CHANGELOG.md`, `LICENSE`)
- Do NOT delete sub-directory READMEs that are auto-generated or consumed by tooling — specifically:
  - `charts/README.md` — often generated by helm-docs and rendered in Artifact Hub/Helm repos; preserve it
  - `test/README.md` — test utility docs consumed by contributors; preserve it
  - Any README inside a sub-package directory that has distinct tooling consumers

**Hosted documentation exception:** If `docs/` contains a site generator config file — `docusaurus.config.js`, `docusaurus.config.ts`, `mkdocs.yml`, `_config.yml` (Jekyll), `conf.py` (Sphinx), `book.toml` (mdBook), or `astro.config.*` (Starlight) — the directory powers a hosted documentation site and must NOT be deleted. In this case:

- Preserve the entire `docs/` directory and its build config
- Still consolidate any *standalone* `.md` files from `docs/` that are not part of the site's content tree (e.g., a stray `docs/OLD_NOTES.md`)
- Add a note in README.md linking to the hosted docs site

The goal: a developer opening this repo sees exactly 4 markdown files at the root (README.md, ARCHITECTURE.md, DEVELOPER_GUIDE.md, CLAUDE.md). No `docs/` subdirectory, with two exceptions: it powers a hosted documentation site, or it contains only diagram files in `docs/diagrams/` (see Diagrams in Step 4). No scattered `.md` files at root. Every fact lives in exactly one place.

## Step 3: Create CLAUDE.md (root level)

Create `CLAUDE.md` at the repo root with these sections (fill from the actual codebase, not placeholders):

- **Project Overview** — 1-2 sentences: what this service/library does and why it exists
- **Key Documentation** — list the 3 docs (ARCHITECTURE.md, DEVELOPER_GUIDE.md, README.md) with ownership rules: what kind of change goes where
- **Build & Run** — table of ALL build/run commands with brief descriptions. **If a Makefile exists, document `make` targets as the primary commands** (e.g. `make test`, `make build`, `make generate`) — do not list raw tool invocations when a Makefile target wraps them. Include prerequisite steps (e.g. `nvm use`, virtualenv activation) before the commands table.
- **Code Generation** — if applicable: table of codegen commands with columns: Command | Input Spec | Output File | Description. Include a warning not to manually edit generated files. List codegen config files separately.
- **Testing** — framework, setup file, test libraries, and run commands. Note any special setup (e2e tokens, local services, environment-specific config, etc.)
- **Project Structure** — tree view of top-level directories with 1-line descriptions
- **Key Conventions** — 5-10 non-obvious rules someone needs to know: path aliases, state management patterns, generated files, central types, auth mechanisms, pre-commit hooks, etc.

## Step 4: Create ARCHITECTURE.md

Create `ARCHITECTURE.md` as the single technical reference. Fill every section from the actual codebase — verify all file paths and function names before writing.

### Identify the repo type

Before writing sections, classify the repo into one of these types. This determines which sections are required:

| Repo type | How to identify | Example |
| --------- | --------------- | ------- |
| **Backend service** | Has HTTP handlers, gRPC servers, queue consumers, database connections | REST API, microservice, worker |
| **Frontend app** | Has `package.json` with React/Vue/Angular/Svelte, HTML entry point | SPA, dashboard, admin panel |
| **Library / SDK** | Exports public API, no main entrypoint or server, published as package | npm package, Go module, pip package |
| **CLI tool** | Has a `main` with arg parsing (cobra, argparse, commander), no long-running server | Developer tools, scripts, automation |
| **Data pipeline** | DAGs, ETL jobs, data transforms, scheduled tasks | Airflow DAGs, Spark jobs, cron scripts |

If a repo spans multiple types (e.g., a monorepo with a backend + frontend), include sections from all applicable types.

### Section details

**All repo types:**

- **§1 Overview** — what the service/library/tool does, mermaid diagram of the main flow (sequence or architecture as appropriate), component map table (Component | Package/Directory | Responsibility)

**Backend service sections:**

- **§2 One section per inbound API** — for each USED endpoint: HTTP method, path, implementation file, request body with full field table (field | type | required | description), every response status code with condition, complete processing flow (numbered steps), auth/middleware chain. Include a full example request JSON payload.
- **§3 One section per outbound integration** — for each external API actually CALLED by the service: HTTP method, endpoint, implementation file, complete request payload with full field table and example JSON, response handling table (every status code → action/error), error classification (retryable vs permanent with error variable names). Do NOT document endpoints from external APIs that this service does not use.
- **§4 Proxy / Gateway Layer** (if applicable) — for services that proxy requests to backends: proxy routing table (Route Pattern | Backend Service | Auth), token exchange or credential injection flows, configuration file locations
- **§5 Security & Auth** (if applicable) — document: cookie security flags (HttpOnly, Secure, SameSite, Path, expiry), CSRF protection mechanism (state param, nonce, etc.), token storage approach (cookie vs header vs memory), session lifecycle (creation, expiry, invalidation), container security (non-root UID, dropped capabilities, read-only filesystem). Reference implementation file for each claim.
- **§6 Event processing / message queues** (if applicable) — for each queue consumed or produced: broker address, topic/queue name, message format with full schema (field table or struct), serialization format (JSON, protobuf, Avro, CloudEvents), consumer group config, auth mechanism, processing pipeline (numbered steps from message receipt to final action), state/event mapping tables, error handling (which errors retry vs skip vs ignore)
- **§7 Data Models** — document ALL types that cross system boundaries (API request/response types, database models, queue message schemas, shared/common types). For each: name, file, full field table (field | type | JSON tag | description). For database models: include table name, column types, indexes, relationships. For queue messages: include the full message envelope and payload structure. Also document: key interfaces (name, file, method signatures), domain constants/labels/enums with their values

**Frontend app sections:**

- **§7 Data Models** — TypeScript types/interfaces for API responses, form data, shared state shapes. For each: name, file, full field table.
- **§8 State Management** — state management approach (Redux, context, stores), store configuration, slice/module structure, API data caching strategy, routing structure

**Library / SDK sections:**

- **§7 Data Models** — exported types, interfaces, enums. For each: name, file, field table with descriptions.
- **§9 Public API / Exports** — table of all exported functions, classes, types, and constants: Name | File | Signature | Description. Group by module if applicable. Document breaking changes policy and versioning scheme.

**CLI tool sections:**

- **§10 Commands & Flags** — table of all commands/subcommands: Command | Description | Flags | Example. Document global flags separately. Include exit codes table (Code | Meaning). Document stdin/stdout/stderr behavior and piping conventions.

**Data pipeline sections:**

- **§6 Event processing / queues** — as above
- **§11 Pipeline / DAG Structure** — DAG/job graph with dependencies (mermaid preferred), schedule/trigger config, input/output datasets table (Dataset | Format | Location | Schema), idempotency and retry behavior, backfill procedure

**All repo types (continued):**

- **§12 Configuration** — grouped tables with columns: Config Key | Env Var | Default | Description. Include separate tables for different config files or environments if applicable.
- **§13 Metrics & Observability** — table: Metric | Description | Tags. Include monitoring dashboards, error tracking (Sentry, etc.), and analytics if applicable.
- **§14 Testing** — table: Type | File/Directory | Description
- **§15 OpenAPI/Proto/GraphQL Specs** (if applicable) — table: Spec | Location | Generated Output
- **§16 References** — external API docs, design docs, meeting notes

Rules:

- **Accuracy is the top priority** — verify every file path, function name, field name, and status code against the actual code before writing. If you cannot verify it, do not include it.
- Do NOT use line numbers in documentation — reference file paths and symbol names only
- Only document endpoints and integrations that are actually USED by the service. Do NOT document available-but-unused endpoints from external API specs.
- Include full example JSON payloads for every API call (inbound and outbound)
- Include full field tables for every request/response body, database model, and queue message schema
- Every section must reference the implementation file

### Diagrams

Use the right format for the complexity:

| Complexity | Format | When |
| ---------- | ------ | ---- |
| Simple (< 10 nodes, linear flow) | Mermaid in markdown | Inline in the doc, renders natively on GitHub |
| Complex (many nodes, crossing arrows, dense layout) | SVG in `docs/diagrams/` | Mermaid becomes unreadable; hand-crafted SVG gives full layout control |
| Both useful | Mermaid + SVG | Mermaid for quick inline reference, SVG for detailed view |

**Mermaid** — use for simple sequence diagrams, basic flowcharts, and small component diagrams that render cleanly inline.

**SVG** — generate a standalone SVG file in `docs/diagrams/` when:

- **Architecture overview** — component boxes with labeled arrows showing data flow
- **Request/response flows** — complex multi-service sequence diagrams
- **Data model relationships** — entity-relationship diagrams with many tables
- **Pipeline/DAG structure** — job dependency graphs with many nodes

Link SVGs from the relevant markdown doc: `![Architecture](docs/diagrams/architecture.svg)`

SVG guidelines:

- Write SVG directly (no external tools required)
- Use a clean, minimal style — white background, thin borders, readable font sizes (14-16px)
- Use `viewBox` for responsive scaling, no fixed `width`/`height`
- Color-code by component type (e.g. blue for services, green for databases, orange for queues)
- Keep text horizontal and legible

## Step 5: Create DEVELOPER_GUIDE.md

Create `DEVELOPER_GUIDE.md` for operational and troubleshooting content. Move any such content from existing docs here.

Required sections:

1. **Troubleshooting** — common issues, diagnosis steps, links to logs/dashboards/monitoring
2. **Auth/Secrets/Security Setup** (if applicable) — credentials, policies, secrets, access control setup, token flows. For file-based secrets (e.g. a `.secrets/` directory convention), document: the expected directory layout, each file name and what credential it contains, how to obtain each credential, and what happens when one is missing or expired.
3. **Environment Setup** (if applicable) — environment variables, config files, templates (e.g. `.env.development.template`), what values to fill in and where to get them
4. **API Examples** (if applicable) — curl examples for testing integrations during development
5. **Release & CI/CD** (if applicable) — document the release pipeline: how versions are bumped (manual tag, auto-increment, GoReleaser, etc.), what triggers a build, where artifacts are published (Artifactory, Docker registry, npm, etc.), and how to trigger a manual release or rollback. Reference CI config files (e.g. `Jenkinsfile`, `.github/workflows/`, `.goreleaser.yaml`).
6. **Enhancement Ideas** — known improvement opportunities

## Step 5b: Update README.md

Restructure README.md to be a concise landing page:

1. **Title line** with language badge
2. **Badge rows** — extract all telemetry/deployment/CI links from existing README into inline badge format (CI status, coverage, deployment, monitoring, etc.). For internal/private CI systems (Jenkins, internal GitLab, etc.), use the badge URL as-is even if it requires VPN/auth to render — do not fabricate a public shields.io badge that misrepresents the actual system.
3. **About section** — 2-3 sentences on what the service does, link to ARCHITECTURE.md
4. **Development section** — prerequisites (Node version, tools), local setup instructions, run commands
5. **Test section** — test commands and special setup
6. **Documentation table** — table linking to ARCHITECTURE.md and DEVELOPER_GUIDE.md with brief descriptions of what each covers

Rules: remove any sections that duplicate ARCHITECTURE.md or DEVELOPER_GUIDE.md. No data flows, interface descriptions, or integration details in README. No troubleshooting, auth/secrets setup, or API examples in README. Replace removed content with links to the appropriate doc.

## Step 6: Clean up

1. Present the list of files to be deleted (consolidated docs, `docs/` directory, standalone docs like API.md, INTEGRATION.md, DATA_MODEL.md, CONTRIBUTING.md) and **ask the user for approval before deleting**
2. Update any cross-references in remaining docs
3. Run `repo-verify-docs` to confirm everything is consistent
4. Run `repo-lint-docs` to fix any markdown issues

## Step 7: Verify

After all changes, confirm:

- `README.md` is a concise landing page with no duplicated content
- `ARCHITECTURE.md` has full technical detail with verified file paths
- `DEVELOPER_GUIDE.md` has all operational/troubleshooting content
- `CLAUDE.md` accurately reflects build commands and project structure
- No fact appears in more than one document
- All cross-doc links work
- All file/function references in docs exist in code
- Build still passes (run the repo's build command to verify)

## Step 8: Offer to commit

**Ask the user before committing.** Do NOT run `git-commit` automatically.

If the user agrees, use the `git-commit` skill to stage and commit all documentation changes. If any scripts were created or modified during this workflow, use the `simplify` skill to review them before committing.
